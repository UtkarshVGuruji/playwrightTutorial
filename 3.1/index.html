    3.1

    Lab 3.1: Overview of Mappings

    Objective:

    In this lab, you are going to work through the process of defining a custom mapping.

    1. Within the Console, index the following sample document, which also creates a new index called sample_blog: 

      POST sample_blog/_doc
{
  "@timestamp": "2021-03-10T16:00:00.000Z",
  "abstract": "The Joy of Painting",
  "author": "Bob Ross",
  "body": "Painting should do one thing. It should put happiness in your heart. We'll take a little bit of Van Dyke Brown. Isn't that fantastic? You can just push a little tree out of your brush like that. Mix your color marbly don't mix it dead.",
  "body_word_count": 55,
  "category": "Painting",
  "title": "Making Happy Little Trees",
  "url": "/blog/happy-little-trees",
  "published": true
}

    2. View the default mappings that were created, and notice that Elasticsearch did its best to guess the data types - but notice a lot of the fields are of type text and keyword: 

      GET sample_blog/_mappings

    3. Create a new index called test_blogs based on the sample_blog mapping, but configure test_blogs to satisfy the following requirements:

      • @timestamp is a date
      • body_word_count is a long
      • the abstract, body, and title fields are of type text only
      • the author, category and url fields are of type keyword only
      • published is of type boolean

      Show answer 

      PUT test_blogs
{
  "mappings": {
    "properties": {
      "@timestamp": {
        "type": "date"
      },
      "abstract": {
        "type": "text"
      },
      "author": {
        "type": "keyword"
      },
      "body": {
        "type": "text"
      },
      "body_word_count": {
        "type": "long"
      },
      "category": {
        "type": "keyword"
      },
      "title": {
        "type": "text"
      },
      "url": {
        "type": "keyword"
      },
      "published": {
        "type": "boolean"
      }
    }
  }
}

    4. Index the document from step 1 into your new test_blogs index. The document should be indexed without any issues with the mapping or data types.

      Show answer 

      POST test_blogs/_doc
{

    5. Now let's make some changes to the blogs index. View the current mapping of blogs: 

      GET blogs/_mappings

    6. Recall the blogs mapping was created when you uploaded the blogs in an earlier lab. The mappings are definitely closer to the kind of optimizations we like to see. For instance, notice that shorter string fields like category are mapped to keyword only, while longer string fields like content are mapped to text only. These mappings were created based on the mappings from the Data Visualizer tool.

    7. The mapping from the Data Visualizer is not perfect though. For example, the author and tags objects contain fields that are mapped as both text and keyword. This is not very efficient, as most of these fields will not need both. Let's make some changes in a new index mapping.     9. Now you can run the PUT command and create the new blogs_fixed index. It will have no data in it yet, but you will do that next.

    10. EXAM PREP: Reindex all of the documents in the blogs index into your new blogs_fixed index.

      Show answer 

      POST _reindex
{
  "source": {
    "index": "blogs"
  },
  "dest": {
    "index": "blogs_fixed"
  }
}

    11. Notice the reindex takes a few moments, but should run fairly quickly. If it times out, do not panic and do not run the reindex command again. It just means it took more than 1 minute, and the Console stopped waiting for the response. Run the following command to see how many documents are in blogs_fixed - you will know the reindexing is complete when blogs_fixed has 4,719 documents. 

      GET blogs_fixed/_search

    12. Let's confirm our changes worked with a simple test. Run these two queries in the console: 

      GET blogs_fixed/_search
{
  "query": {
    "match": {
      "authors.full_name": "kim"
    }
  }
}

GET blogs_fixed/_search
{
  "query": {
    "match": {
      "authors.first_name": "kim"
    }
  }
}
      The first query should have 46 hits, while the second will have zero. This is because keyword fields are not lowercased like text fields.

    13. Search for "Kim" as the authors.first_name and you should get 3 hits.

    14. Write a query on blogs_fixed that hits blogs with "security analytics" as a value in tags.use_case. You should get 216 hits.

      Show answer 

      GET blogs_fixed/_search
{
  "query": {
    "match": {
      "tags.use_case": "security analytics"
    }
  }
}

    15. Run your previous query for "security analytics" on the original blogs index. You should get 598 hits. Why are there so many more hits?

      Show answer The tags.use_case field in the blogs index is of type text, so searching for "security analytics" is a search for "security" or "analytics", so you are hitting blogs that also have "business analytics" as a use case. When you ran the same search on the blogs_fixed index, you hit blogs that have the exact use case of "security analytics" - which demonstrates the difference between the text and keyword fields. You will see an explanation of this behavior later in this module.

    Summary:

    In this lab, you created a new index that improves on the mapping from the original blogs mapping that was created from the Data Visualizer. We will still have some improvements to make on blogs, but for now you should be familiar with the process of defining a mapping for an index.
